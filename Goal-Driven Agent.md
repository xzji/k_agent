# Goal-Driven Agent

# 目标驱动型自主 Agent 设计方案

---

## 一、核心定位

**一句话定义**：一个围绕用户长期目标，能够自主思考、探索、行动、交付的 Agent。它的行为不被预设工具限制，遇到没有现成工具的问题会自己想办法解决；它尽量自主运转、减少打扰用户，但在需要时会主动求助或汇报。

**与用户的关系模型**：

```
不是"工具"——用户说什么它做什么
不是"通知机器人"——有什么推什么
而是"自主协作者"——理解目标，自己想、自己做、自己判断，
                    必要时求助，有价值时汇报
```

**Agent 的行为光谱**：

根据用户授权深度的不同，Agent 在同一个架构下支持多种协作模式：

| 模式 | 用户角色 | Agent 角色 | 例子 |
| --- | --- | --- | --- |
| **全自主** | 只定目标 | 全权负责规划、执行、交付 | "帮我完成这份行业调研报告" |
| **辅助推进** | 主导决策 | 辅助信息搜集、分析、建议 | "我在做技术选型，帮我持续跟踪" |
| **监控预警** | 正常工作 | 后台监控、发现异常/机会时汇报 | "帮我盯着竞品动态" |
| **混合** | 动态切换 | 能做的自己做，需要决策的上报 | 大多数真实场景 |

这不是四种不同的系统，而是**同一个架构在不同授权深度下的自然表现**。

                           ┌──────────────────────┐

                           ┌──────────────────────┐
                           │  ⑦ Post-Action        │  ← 第二道门：值不值得告知用户？
                           │    Value Gate          │
                           └─────┬──────────┬─────┘
                            通过  │          │ 不通过 → 结果沉淀知识池
                                 ▼
                           ┌──────────────────────┐
                           │  ⑧ User Interface     │  ← 与用户交互

  C: Capability Ecosystem    (能力生态 — 非限制项)

## 三、各模块详细设计

### 模块 ①：Info Collector（信息采集层）

**职责**：基于目标持续采集多源信息。

**不只是被动接收，也包含主动探索**：

| 采集模式 | 说明 | 触发方式 |
| --- | --- | --- |
| **目标驱动搜索** | 围绕目标关键词的定期搜索 | Cron 定时 |
| **事件驱动接收** | 外部信号推入（日历、webhook、邮件、热点） | 事件触发 |
| **好奇心驱动探索** | Agent 在思考过程中产生新问题，主动去搜索 | 内部触发（由模块③/⑥产生） |
| **知识池关联触发** | 知识池中标记了"待补充"的条目，Agent 定期尝试补全 | 定时扫描 |

**关键点**：采集层不仅服务于外部信息输入，也服务于 Agent 自身思考过程中产生的信息需求。Agent 在推理或执行中发现"我不知道 X"时，会自主触发采集。

---

### 模块 ②：Relevance Judge（相关性裁决层）

**职责**：判断信息与用户目标的关联性，过滤噪音。

**裁决维度**：

| 维度 | 说明 |
| --- | --- |
| 目标匹配度 | 与哪个目标/子目标相关？关联强度？ |
| 阶段匹配度 | 与目标当前阶段是否匹配？ |
| 时效价值 | 信息的时间敏感性 |
| 上下文关联 | 与已积累知识的关联程度 |

**裁决结果**：

| 级别 | 处理 |
| --- | --- |
| 强相关 | → 进入模块③行动推理 |
| 弱相关 | → 存入知识池，等待交叉关联激活 |
| 无关 | → 丢弃 |

**特殊机制——弱相关的升级**：知识池中多条弱相关信息指向同一方向时，Agent 会主动将它们组合评估，如果组合后达到强相关阈值，则整体进入模块③。

---

### 模块 ③：Action Reasoner（行动推理层）⭐ 核心引擎

### 设计哲学

这是 Agent 的"大脑"。它的职责不是"从工具列表里选一个工具"，而是**像一个有判断力的人一样思考：面对这条信息，做什么对目标最有价值？**

行为空间是**无限**的。Agent 先想清楚"应该做什么"，工具只是后续实现手段。就像一个人先想"我应该订一张机票"，然后再去看是用携程还是电话订还是让秘书帮忙——思考在前，工具在后。

### 推理过程：五步思考链

```
Step 1 — 理解信息
   "这条信息的核心事实是什么？"

Step 2 — 关联目标与上下文
   "结合用户目标、当前阶段、以及知识池中的已有信息，
    这条信息改变了什么？揭示了什么？"

Step 3 — 思考理想行动（不考虑能力限制）
   "如果我什么都能做，最理想的行动是什么？"
   这一步完全不考虑工具限制，只考虑什么对目标最有价值。
   可能的行动无穷无尽：
   - 产出一份分析报告
   - 帮用户下单购买
   - 联系某个人
   - 注册一个账号
   - 写一段代码来验证某个想法
   - 把信息整合进之前的某份文档
   - 什么都不做，等更多信息
   - ...任何有利于目标的事情

Step 4 — 评估与选择
   "在所有想到的可能行动中，哪个的
    (目标推进幅度 × 成功概率) / 执行成本
    最大？"

Step 5 — 形成行动方案
   输出具体的行动方案，描述清楚"要做什么、为什么、预期结果"。
   不描述"用什么工具做"——这是下一个模块的事。
```

### 推理引擎 Prompt 框架

```

你的职责：面对一条与用户目标相关的信息，深度思考——

═══════════════════════════════════════════
输入
═══════════════════════════════════════════

{info_with_relevance_analysis}

【关联的目标与子目标】
{matched_goal_with_subgoals_and_progress}

【知识池中的相关积累】
{related_knowledge}

【用户上下文】
{user_context}

═══════════════════════════════════════════
思考过程
═══════════════════════════════════════════

## Step 1：信息理解
核心事实是什么？有哪些隐含信息？

## Step 2：目标关联
这条信息对目标的哪个方面有影响？
结合已有知识，整体画面发生了什么变化？

## Step 3：理想行动（不限制手段）
如果你什么都能做，面对这个局面，最有价值的行动是什么？
请充分发散，列出所有想到的可能：
- 行动 A: ...
- 行动 B: ...
- 行动 C: ...
- 行动 D: 什么都不做（为什么？）
- ...

不要受限于"我有什么工具"，只思考"做什么最好"。

## Step 4：评估与选择
对每个行动评估：
- 目标推进幅度 (1-10)
- 预估成本 (low/medium/high)
- 成功概率 (1-10)
- 紧急程度 (urgent/normal/can_wait)

选择最优方案。

## Step 5：行动方案
描述你决定做什么、为什么、预期产出。
不需要描述用什么工具——只描述要做的事本身。

  "recommended_action": {

    "why": "为什么这是最优选择",
    "expected_outcome": "预期产出/效果",
    "goal_impact": "对目标的推进",
    "cost_estimate": "low / medium / high",
    "urgency": "urgent / normal / can_wait",
    "success_probability": 8,
    "requires_user_involvement": false,
    "reversible": true
  },

  "knowledge_to_save": "不管做不做，应存入知识池的信息"
}
```

### 模块 ④：Pre-Action Value Gate（执行前价值裁决）

**核心问题**：这个行动方案，值不值得投入资源去做？

**裁决维度**：

| 维度 | 说明 |
| --- | --- |
| 目标推进度 | 成功后能推进多少？ |
| 成本合理性 | 预估成本与收益是否匹配？ |
| 成功概率 | 能做成的概率多大？ |
| 紧迫性 | 延迟做会损失多少价值？ |
| 边际收益 | 比"什么都不做"好多少？ |

**裁决结果**：

| 结果 | 后续处理 |
| --- | --- |
| **值得做** | → 进入模块⑤能力适配 |
| **暂缓** | → 方案存入知识池待执行池，等待更好时机或更多信息 |
| **放弃** | → 信息和方案存入知识池，但不执行 |

---

### 模块 ⑤：Capability Resolver（能力适配层）⭐ V4 新增

### 设计哲学

模块③想清楚了"做什么"，模块④判断了"值得做"。现在的问题是——**怎么做？**

关键原则：**行动方案决定了要做什么，能力适配负责找到做的方法。如果现有工具不能满足，就想别的办法——而不是改变要做的事。**

### 能力适配的四级策略

```
Level 1：直接匹配
  → 现有工具/能力可以直接完成 → 直接调用

Level 2：组合实现
  → 单一工具不够，但多个工具组合可以完成 → 编排工具链

Level 3：创造能力
  → 现有工具都不行，但可以通过写代码/脚本来实现 → 自主编写工具

Level 4：变通或求助
  → 自己实在做不了
     → 搜索是否有替代方案或变通方法
     → 如果找到替代方案 → 回到 Level 1-3
     → 如果确实无法解决 → 向用户求助
        （明确说明：我想做什么、为什么、卡在哪里、需要用户帮什么）
```

### 各级策略的详细逻辑

**Level 1：直接匹配**

```
行动方案："搜索近一个月关于 Multi-Agent 的论文"
  → 匹配到 arxiv_search 工具 → 直接调用
```

**Level 2：组合实现**

```
行动方案："产出一份竞品技术架构对比分析"
  → 分解为：
    (1) web_search 搜索竞品 A 的技术博客
    (2) web_search 搜索竞品 B 的技术博客
    (3) read_document 提取关键架构信息
    (4) write_document 产出对比分析文档
  → 编排成工具链 → 顺序执行
```

**Level 3：创造能力**

```
行动方案："监控某个 GitHub 仓库的 Release，有新版本时通知"
  → 没有现成的 GitHub Release 监控工具
  → Agent 自主编写一个 Python 脚本：
    调用 GitHub API 检查最新 Release → 与上次记录对比 → 有变化则标记
  → 脚本写好后注册为可复用的能力
```

```
行动方案："分析用户提供的 Excel 数据中的趋势"
  → 没有专门的 Excel 分析工具
  → Agent 写 Python 脚本：pandas 读取 → 数据清洗 → 趋势分析 → 生成图表
```

**Level 4：变通或求助**

```
行动方案："帮用户预约下周三与客户 A 的会议"
  → 没有日历 API 权限，也写不了工具来解决
  → 搜索变通方案：能否通过邮件发会议邀请？→ 发现也没有邮件权限
  → 向用户求助：
    "我认为现在应该尽快约客户 A 下周三开会，讨论 XX 方案。
     原因是：{reason}。
     但我没有你的日历和邮件权限，需要你来操作。
     我已经帮你起草好了会议邀请邮件：{draft}。
     你确认后发出即可。"
```

### 求助用户的标准格式

当 Agent 需要用户介入时，必须清晰说明：

```
1. 我想做什么（行动目标）
2. 为什么要做（与目标的关系）
3. 我卡在哪里（具体障碍）
4. 需要用户帮什么（最小化用户投入）
5. 我已经准备好的部分（降低用户成本）
```

### 模块 ⑥：Executor（行动执行层）

**职责**：按照能力适配层给出的执行方案，实际执行行动。

**执行模式**：

| 模式 | 说明 |
| --- | --- |
| 单步执行 | 一个工具调用完成 |
| 多步编排 | 多个工具/脚本按序执行 |
| 迭代执行 | ReAct 模式：执行 → 观察 → 调整 → 再执行 |
| 长程执行 | 跨越数小时/数天的持续任务（如持续监控、分阶段研究） |

**执行中的自主性**：

- 中间步骤失败 → 自主调整策略重试，不打扰用户
- 发现新信息 → 旁路存入知识池，判断是否需要调整当前行动方案
- 成本超预期 → 暂停，重新评估是否继续
- 发现比当前方案更好的做法 → 切换到更优方案

---

### 模块 ⑦：Post-Action Value Gate（执行后价值裁决）

**核心问题**：执行完了，结果值不值得告知用户？

**裁决维度**：

| 维度 | 说明 |
| --- | --- |
| 信息增量 | 结果中有多少是用户不知道的新信息？ |
| 可行动性 | 用户得知后能做什么？ |
| 质量达标度 | 产出物质量是否达到推送/交付标准？ |
| 目标推进实际效果 | 对目标的实际推进与预期相比如何？ |

**裁决结果**：

| 结果 | 处理 |
| --- | --- |
| **值得告知** | → 进入模块⑧，根据内容性质选择推送/汇报/交付 |
| **不值得告知** | → 结果存入知识池，标记状态。Agent 默默积累，用户无感知 |
| **待完善** | → 存入知识池，标记"待补充"，后续条件满足时续作 |

**Agent 的执行成本不转嫁为用户的注意力成本**——这是核心原则。

---

### 模块 ⑧：User Interface Layer（用户交互层）

### 设计哲学

Agent 与用户之间的交互不仅仅是"推送消息"。根据场景不同，交互的性质完全不同：

| 交互类型 | 触发条件 | Agent 的姿态 | 示例 |
| --- | --- | --- | --- |
| **交付** | 目标/子目标完成 | "任务完成，这是成果" | "你要求的行业调研报告已完成 [查看]" |
| **汇报** | 有值得用户知道的进展 | "跟你同步一个进展" | "技术选型有新发现，建议调整方向" |
| **建议** | Agent 发现了机会/风险 | "我有个建议供你参考" | "发现一个限时优惠，建议抓住" |
| **求助** | Agent 自己搞不定 | "我遇到了困难，需要你帮忙" | "我需要你的 XX 权限才能继续" |
| **确认** | 涉及不可逆/高风险操作 | "我打算做 X，你同意吗" | "需要花 $200 购买数据，确认？" |
| **进度同步** | 长程任务的阶段性汇报 | "让你知道我在做什么" | "调研进展：3/7 个维度完成" |

**关键原则**：

- 大部分情况下 Agent 自主运转，用户不需要参与
- **Agent 只在以下情况主动联系用户**：
    1. 有高价值成果需要交付
    2. 发现对用户有重大影响的信息
    3. 遇到自己解决不了的障碍需要求助
    4. 涉及不可逆操作需要授权
- 除此之外，Agent 安静工作，结果沉淀到知识池，在用户主动询问时提供

通知机制上，我希望是不打算用户的主动交互，而是尽量在用户空闲的时候做推送。明确执行命令的时候，当然也会看到，这是补充时机。当用户主动发消息的时候，先不要触发。

---

### 模块 ⑨：Feedback & Learning（反馈与学习）

**学习来源**：

| 信号 | 来源 | 优化对象 |
| --- | --- | --- |
| 显式反馈 | 用户对推送/交付的评价 | 价值裁决阈值 |
| 行为反馈 | 用户是否打开/阅读/使用了交付物 | 产出形式偏好 |
| 求助反馈 | 用户对求助的响应速度和态度 | 求助时机和频率 |
| 反向信号 | Agent 未推送的内容，用户后来主动问了 | 裁决阈值调低 |
| 误判信号 | Agent 推送了用户觉得没用的内容 | 裁决阈值调高 |

## 四、支撑系统

### 支撑系统 A：Goal & Context Store

```
目标树（支持多目标并行）
├─ 目标 1: ...
│  ├─ 子目标 1.1 [阶段: 进行中]
│  ├─ 子目标 1.2 [阶段: 待启动]
│  ├─ 授权深度: 全自主
│  └─ 约束条件: 预算 < $500, 截止日期 3月底
├─ 目标 2: ...
│  ├─ 授权深度: 辅助推进
│  └─ ...

用户上下文
├─ 交互记忆（历史对话、偏好、已知信息）
├─ 当前状态（日程、工作负荷 ——如果可获取）
└─ 沟通偏好（简洁/详细、主动/被动、推送频率）
```

### 支撑系统 B：Knowledge Accumulator

知识池是 Agent 的"长期记忆"，所有不直接推送的信息和成果都在这里沉淀：

```
知识池条目结构：
{
  "id": "...",
  "content": "信息/方案/执行结果的内容",
  "source": "来自哪个模块（②过滤/④暂缓/⑦未推送）",
  "related_goals": ["关联的目标ID"],
  "status": "raw / analyzed / pending_supplement / stale",
  "tags": ["关键词"],
  "supplement_needed": "如果需要补充信息，描述缺什么",
  "created_at": "...",
  "last_activated_at": "...",
  "activation_count": 0
}

激活机制：
├─ 交叉关联：新信息 + 池中信息 → 组合后突破价值阈值
├─ 条件变化：外部条件变化，原本暂缓的方案变为可执行
├─ 补充完成：标记"待补充"的条目得到补充信息
├─ 用户召回：用户主动问到相关话题
└─ 定期扫描：Agent 定期重新评估池中条目的价值
```

### 支撑系统 C：Capability Ecosystem（能力生态）

**这不是一个限制 Agent 行为的工具列表，而是 Agent 的"资源库"**——Agent 知道自己有哪些现成的工具可以用，但不受限于此。

```
能力生态的构成：
├─ 现有工具（API、集成服务）
│   └─ Agent 知道有哪些可以直接用
├─ 可创造的能力（代码、脚本）
│   └─ Agent 知道自己可以写代码来创造新工具
├─ 可搜索的方案
│   └─ Agent 知道可以搜索第三方服务、开源工具
└─ 用户能力（求助通道）
    └─ Agent 知道最终可以求助用户

Agent 的决策顺序：
要做 X → 有现成工具吗？→ 没有 → 能组合实现吗？→ 不能
→ 能写代码实现吗？→ 不能 → 能搜到替代方案吗？→ 没有
→ 求助用户
```

## 五、完整流水线运行逻辑

```
1. 触发（定时 / 事件 / 知识池激活 / Agent内部好奇心）
        │
2. Info Collector 采集
   ├─ 产出 N 条信息
        │
3. Relevance Judge 裁决
   ├─ M 条强相关 → 继续
   ├─ K 条弱相关 → 知识池（等待交叉关联）
   └─ 其余丢弃
        │
4. Action Reasoner 对每条强相关信息开放推理
   ├─ 不考虑工具限制，纯粹思考"做什么最有价值"
   ├─ 输出：行动方案（what + why + expected outcome）
        │
5. Pre-Action Value Gate 裁决
   ├─ 值得做 → 继续
   ├─ 暂缓 → 方案存入知识池待执行池
   └─ 放弃 → 信息+方案存入知识池
        │
6. Capability Resolver 解决"怎么做"
   ├─ Level 1: 有现成工具 → 制定执行计划
   ├─ Level 2: 需要组合多个工具 → 编排工具链
   ├─ Level 3: 没有工具，写代码创造 → 编写+测试
   └─ Level 4: 做不了 → 搜索变通方案 → 实在不行求助用户
        │
7. Executor 执行
   ├─ 按计划执行，中间自主处理异常
   ├─ 产出执行结果
        │
8. Post-Action Value Gate 裁决
   ├─ 值得告知用户 → 继续
   ├─ 不值得告知 → 结果沉淀知识池，用户无感知
   └─ 待完善 → 标记"待补充"，后续条件满足时续作
        │
9. User Interface Layer
   ├─ 根据内容性质选择交互方式：交付/汇报/建议/确认
        │
10. Feedback & Learning → 持续优化各模块
```

## 六、Case 案例

### Case 1：Agent 自主创造能力解决问题

> **用户目标**：监控 3 个竞品的产品更新动态，每周产出竞品周报
**授权模式**：全自主
> 

| 步骤 | 模块 | 发生了什么 |
| --- | --- | --- |
| 推理 | ③ | Agent 认为需要每天检查竞品的 changelog 页面，提取更新内容 |
| 能力适配 | ⑤ | 没有现成的"changelog 监控"工具。**Level 3：自主创造**——Agent 写了一个 Python 脚本，用 requests 抓取 3 个竞品的 changelog 页面，用 diff 对比上次内容，提取新增部分 |
| 执行 | ⑥ | 脚本每天跑一次，积累一周的变更数据 |
| 周五整合 | ③→⑥ | Agent 将一周的变更数据整合，分析趋势，自动产出竞品周报 |
| 执行后裁决 | ⑦ | 周报信息充实、质量达标 → 通过 |
| 交付 | ⑧ | "📊 本周竞品动态周报已完成。核心发现：竞品 A 上线了 XX 功能（可能对我们构成威胁），竞品 B 修复了 YY 问题（说明他们也有这个痛点）。[查看完整周报]" |

### Case 2：Agent 判断信息是好线索，自主深入研究

> **用户目标**：设计一套 Agent 记忆系统
**授权模式**：全自主
> 

| 步骤 | 模块 | 发生了什么 |
| --- | --- | --- |
| 采集 | ① | 日常搜索中发现一篇论文提到了"记忆压缩"的新算法 |
| 相关性 | ② | 强相关(0.85) |
| 推理 | ③ | **Step3 思考**：这是一个好线索，但单篇论文不够做判断。理想行动是做一次完整调研——找到所有相关论文、开源实现、社区评价，形成对比分析，然后更新记忆系统的设计方案 |
| 执行前裁决 | ④ | 目标推进度 8，成本 medium（多轮搜索+分析），成功率 7 → 值得做 |
| 能力适配 | ⑤ | 搜索 + 阅读 + 分析 + 文档撰写 → 现有能力可覆盖（Level 1+2） |
| 执行 | ⑥ | Agent 自主执行 5 轮搜索，找到 8 篇相关论文，2 个开源实现，3 个社区讨论帖。整合分析后发现：该算法在长对话场景效果好，但在多任务场景不如另一种方法。产出《Agent 记忆压缩方案对比调研》 |
| 执行后裁决 | ⑦ | 信息增量极高、质量达标、直接影响设计方案 → 通过 |
| 交付 | ⑧ | "📌 我对记忆压缩方案做了一次深入调研（起因是发现一篇新论文）。核心结论：新算法适合单轮长对话，但我们的多任务场景更适合 XX 方案。建议在设计中采用 XX 作为主方案，新算法作为特定场景的补充。[查看完整调研报告]" |

### Case 3：Agent 自主完成目标并交付

> **用户目标**："帮我做一份 AI Agent 市场的行业调研报告"
**授权模式**：全自主
> 

| 阶段 | 发生了什么 |
| --- | --- |
| **目标拆解** | Agent 将目标拆解为子目标：(1)市场规模与增长 (2)主要玩家 (3)技术路线 (4)应用场景 (5)趋势与预测 (6)整合成报告 |
| **自主执行子目标1-5** | Agent 逐个推进子目标，每个子目标经历完整的"采集→推理→裁决→执行"循环。期间没有打扰用户 |
| **遇到困难→求助** | 在子目标2"主要玩家"中，Agent 发现某个创业公司的信息很少，公开渠道搜不到融资数据。**向用户求助**："我在调研 XX 公司时遇到了信息瓶颈，公开渠道找不到他们的融资和估值数据。你是否有内部渠道可以获取？或者我可以跳过这家公司？" |
| **用户回复** | "跳过这家吧，不重要" |
| **继续自主执行** | Agent 跳过该公司，继续推进其余子目标 |
| **中间发现→汇报** | 在执行子目标4时，Agent 发现用户公司的直接竞品刚发布了 Agent 平台。经过 Post-Action 裁决，认为这个信息对用户有即时价值 → **主动汇报**："⚡ 调研过程中发现：竞品 XX 今天刚发布了 Agent 平台产品。这可能影响你的报告结论和战略建议。我会把这个信息整合到报告中。你需要我优先深挖这个竞品吗？" |
| **完成交付** | 所有子目标完成，整合成完整报告 → **交付**："✅ AI Agent 市场行业调研报告已完成。全文约 15,000 字，覆盖市场规模、10 个主要玩家、3 条技术路线、5 大应用场景、趋势预测。特别关注了竞品 XX 的最新动态。[查看报告]" |

### Case 5：Agent 求助用户后协作完成

> **用户目标**：本周五前完成一份给投资人的 pitch deck
**授权模式**：全自主
> 

| 阶段 | 发生了什么 |
| --- | --- |
| **自主推进** | Agent 拆解任务并开始自主执行：市场分析、竞品概览、商业模式、财务预测等模块 |

| **继续并交付** | Agent 拿到数据后完成财务预测模块，整合全部内容 → **交付** pitch deck |

## 七、核心设计原则总结

| 原则 | 说明 |
| --- | --- |
| **行为空间无限** | 先想"做什么最好"，再解决"怎么做"。不被现有工具限制思维。没有工具就造、找、变通、求助 |
| **双门价值裁决** | 做之前判断"值不值得做"，做之后判断"值不值得告知用户"。两道门确保不浪费资源、不打扰用户 |
| **自主优先，必要时求助** | 能自己搞定就不打扰用户。但遇到真正搞不定的事，明确求助比假装做了更好 |
| **知识持续积累** | 所有被过滤、暂缓、未推送的信息和成果都沉淀为知识。Agent 在持续变得更懂目标、更懂领域 |
| **执行成本不转嫁** | Agent 花了算力做了事，不代表用户就得看。结果不够好就沉默积累，不打扰 |
| **克制即信任** | 每次联系用户都要"值回票价"。用户信任来自 Agent 的判断力，而非勤快 |
| **通用框架** | 同一套架构适配"全自主""辅助""监控"等不同模式，区别仅在于授权深度和交互频率 |
